# Обзор кода и возникшие вопросы

## Общая оценка

В целом код рабочий. Не без огрехов. Из очевидных вещей в первую очередь нужно упомянуть необходимость тестов и предположение что Add всегда имеет два входа. Так же оптимизация и печать выражений не печатают скобки и оптимизируют только то что было в примере. Но для прототипа кажется это дожно быть ок.

## Сборка и запуск

Проект использует библиотеки pugixml (тестировалось на версии 1.14), boost (1.83), 
prettyprint ( https://github.com/afranchuk/prettyprint/tree/master ), систему сборки cmake (3.28.3), gcc (14.2.0). Сборка тестировалась под Ubuntu 24.04.4 (Для сборки на убунту вероятно нужно поставть пакет build-essential).

Для сборки нужен доступ к github.com (чтобы cmake скачал prettyprint).

### Сборка

```bash

mkdir c-gen && cd c-gen
git clone https://github.com/andreykl/c-gen/ .
mkdir build && cd build
cmake ../
make

```

### Запуск

После сборки в текущем каталоге (c-gen/build) появится файл c-gen и его можно запустить,
передав ему путь к файлу xml:

```bash

./c-gen ../testfile.xml

```

программа выведет результат своей работы на экран

Комментарии и возникшие вопросы ниже.

### 1. Количество выходов у блоков

Считаем, что у всех блоков кроме Outport по одному выходу и выход имеет номер 1, что согласуется с данными в линиях.

### 2. Неуказанные входы

Если у элемента не указан вход, считаем что его номер 1 и знак `+`.

### 3. Печать кода и порядок обхода

При печати кода у меня некоторое несоответствие примеру кода, данному в документе, но это вероятно зависит от порядка обхода нод. Инварианты соблюдены, т.е. мы вычисляем сначала те ноды, значения которых нужны для того чтобы вычислить текущую ноду (те, что предшествуют текущей ноде в графе).

**Алгоритм** — обход графа в глубину, начиная с конца. Когда встречаем `delay` — разрываем цикл, саму ноду `delay` вычисляем просто после всех шагов (если нод `delay` несколько, то они все идут после шагов).

### 4. Трактовка записи номеров портов

У блока с SID 17 поле `Ports` имеет значение `[2, 1]`, а поле `Inputs` — `"+-"`. Я это трактую как два порта: один называется "1" и имеет знак "-", а другой называется "2" и имеет знак "+" (просто подряд).

Ниже идёт Line `"feedback"` из `18:1` в `17:2`. Порт `17:2` по этой логике имеет знак `+`. Но на чертеже и в коде `feedback` имеет знак `-`. Поэтому я трактую надпись `<P Name="Dst">17#in:2</P>` как "в элемент номер 17 в порт по индексу 2", считая что индексы начинаются с единицы. Тогда получается, что мы должны взять порт с номером "1" из массива `[2,1]` и соответствующий ему знак из массива `"+-"` тоже под индексом 2, т.е. `"-"`. Номер порта я игнорирую, беру только знак.

### 5. UnitDelay: SampleTime и начальное значение

У блока `UnitDelay` есть значение `SampleTime = -1`, а начальное значение в коде `0` (в функции `nwocg_generated_init`). Я просто игнорирую `SampleTime` (читаю, заполняю структуру, но игнорирую при генерации кода), и `0` захардкожен, поскольку не ясно, откуда его брать.

### 6. Inport и блок Port

У блока `Inport` есть такой блок:

```xml
<Port>
  <P Name="PortNumber">1</P>
  <P Name="Name">setpoint</P>
</Port>
```

Внутри ноды Port у ноды P с атрибутом Name="Name" значение совпадает с именем, данным в атрибуте Name соответствующей ноды BlockType. Это же имя используется в именах полей в коде Си. Не ясно:

* могут ли эти поля не совпадать;
* если могут, то какое брать;
* каково назначение атрибута PortNumber поля P.
